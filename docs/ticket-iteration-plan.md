# 工单系统迭代计划

本文档用于定义工单系统接下来的分阶段迭代计划，目标是在当前已经具备工单基础能力的前提下，按“先打基础、再做工作台、再做运营化、最后做协作与 AI”的顺序，推进工单系统从 MVP 演进为真正可高频使用、可管理、可扩展的内部客服工单平台。

本文档重点覆盖：

- 迭代阶段划分
- 每个阶段的目标与范围
- 后端 / 前端任务拆分
- 风险点
- 验收标准
- 推荐推进顺序

## 1. 总体目标

当前工单系统已经完成：

- 工单主表建模
- 工单创建与会话转工单
- 工单列表与详情
- 指派 / 状态流转
- 回复 / 内部备注
- 关闭 / 重开
- 关注人与事件日志
- 基础 SLA 记录

下一阶段的总体目标不是继续做零散补丁，而是完成以下四步：

1. 让工单拥有稳定的结构化配置底座
2. 让列表页和详情页成为真正的工作台
3. 让 SLA 和队列具备运营化能力
4. 让工单逐步具备协作和 AI 辅助能力

## 2. 迭代原则

在整个迭代过程中，应遵循以下原则：

- 优先做不会返工的底座能力
- 每个迭代都要形成独立可上线的增量价值
- 避免一开始做过重的通用平台能力
- 优先做最影响客服日常效率的能力
- 优先做最影响主管管理视角的能力
- 规则、AI、报表等能力必须建立在结构化字段和工作台之上

## 3. 阶段划分总览

建议按以下阶段推进：

- Iteration 1：配置中心与结构化表单
- Iteration 2：列表页队列化与批量操作
- Iteration 3：详情页重构与数据权限
- Iteration 4：SLA 风险运营
- Iteration 5：自动化规则与快捷动作
- Iteration 6：协作关系与 AI 辅助

## 4. Iteration 1：配置中心与结构化表单

## 4.1 目标

让工单不再只是带状态的文本记录，而是一个拥有标准配置、标准字段和可运营结构的正式业务对象。

## 4.2 范围

本期包含：

- 工单分类配置
- 工单解决码配置
- 工单 SLA 配置
- 新建/编辑工单弹窗接入分类
- 状态流转接入解决码
- SLA 初始化改为配置驱动

本期不包含：

- 动态字段设计器
- 通用规则引擎
- 客户端字段配置

## 4.3 后端任务

- 新增 `TicketCategory`
- 新增 `TicketResolutionCode`
- 新增 `TicketSLAConfig`
- 新增对应 repository / service / controller / builder
- 注册 model 和 generator
- 增加工单配置权限点
- `CreateTicket` 接入分类校验
- `ChangeStatus` 接入解决码校验
- `initSLAs` 从配置读取目标值，替换硬编码逻辑

## 4.4 前端任务

- 新增分类管理页
- 新增解决码管理页
- 新增 SLA 配置页
- 升级工单创建/编辑弹窗
- 升级状态变更弹窗

## 4.5 风险点

- 现有 SLA 初始化为硬编码，改为配置驱动时需保留兜底策略
- 历史工单无分类、无解决码时，前端需要兼容空值
- 如果本期同时引入过多可配项，会拖慢节奏

## 4.6 验收标准

1. 后台可维护分类、解决码、SLA 配置
2. 新建工单可选择分类
3. 解决工单可填写解决码
4. 新工单 SLA 初始化来自配置
5. 历史工单在新版下可正常查看

## 5. Iteration 2：列表页队列化与批量操作

## 5.1 目标

让工单列表页从 CRUD 表升级为队列工作页，支持高频筛选、快速处理和主管管理。

## 5.2 范围

本期包含：

- 列表页快捷视图改造
- 筛选项扩充
- 表格列优化
- 行级快捷操作
- 批量操作第一版
- summary 指标扩充

本期不包含：

- 保存视图
- 宏 / 模板
- 报表中心

## 5.3 后端任务

- 扩展 `ticket/list` 支持更多筛选
- 增加 summary 指标：
  - `unassigned`
  - `pendingInternal`
  - `overdue`
- 增加批量接口：
  - 批量指派
  - 批量改状态
  - 批量关注
- 为列表结果补充分类型名称和 SLA 风险字段

## 5.4 前端任务

- 工单列表页重构为队列页
- 增加默认快捷视图：
  - 全部
  - 我的
  - 待分配
  - 待客户反馈
  - 待内部处理
  - 已超时
- 增加筛选项：
  - 分类
  - 优先级
  - 严重度
  - 团队
  - 处理人
  - 来源
- 增加表格多选
- 增加批量操作工具条
- 增加 SLA 风险列

## 5.5 风险点

- 列表筛选项变多后，页面复杂度会上升，需要控制布局密度
- 批量操作需要严格保证权限校验和事件留痕一致
- 列表页如果同时承载过多动作，会影响可读性

## 5.6 验收标准

1. 客服可快速看到自己的工单队列
2. 主管可看到待分配与超时队列
3. 列表页支持批量指派和批量改状态
4. 列表页能一眼看出 SLA 风险

## 6. Iteration 3：详情页重构与数据权限

## 6.1 目标

让工单详情页成为主处理场所，同时建立工单数据可见范围控制。

## 6.2 范围

本期包含：

- 工单详情页重构
- 详情接口聚合增强
- 来源会话信息增强
- 关联工单区域预留
- 数据权限第一版

本期不包含：

- 工单协作关系编辑
- 客户端消息回流
- 复杂 ACL 配置

## 6.3 后端任务

- 扩展 `ticket/detail` 聚合信息：
  - 分类名
  - 创建人
  - 来源会话摘要
  - 关联工单列表
- 在 `ticket/list` 和 `ticket/{id}` 接入数据权限控制
- 支持数据范围：
  - 仅本人负责
  - 本团队
  - 我关注的
  - 我创建的
  - 全部工单

## 6.4 前端任务

- 工单详情页重构为双栏布局
- 主区域改为：
  - 回复记录
  - 事件记录
- 侧栏增加：
  - 基础信息
  - 客户信息
  - 指派信息
  - SLA 信息
  - 来源会话
  - 关联工单
- 详情页增强头部信息与操作区

## 6.5 风险点

- 详情页组件拆分不合理会导致状态管理混乱
- 数据权限如果规则定义不清晰，后续容易返工
- 来源会话聚合字段增加后，接口可能变重，需要控制查询方式

## 6.6 验收标准

1. 工单详情页可作为主要处理页
2. 客服处理一张工单不需要频繁回列表
3. 主管可在详情页快速看清工单卡点
4. 数据权限符合角色预期

## 7. Iteration 4：SLA 风险运营

## 7.1 目标

让 SLA 不再只是展示字段，而成为真正驱动工单优先级和主管管理的机制。

## 7.2 范围

本期包含：

- SLA 扫描任务
- SLA breach 标记
- 风险工单视图
- 超时事件日志
- 主管风险看板基础版

本期不包含：

- 复杂升级链
- 邮件 / 短信 / 外部通知
- 客户端 SLA 提示

## 7.3 后端任务

- 新增 SLA 扫描调度任务
- 更新 `TicketSLARecord` 状态为 `breached`
- 写入 `sla_breached` 事件
- 暴露 risk 字段给列表与详情接口
- 增加主管风险统计接口

## 7.4 前端任务

- 列表页支持“即将超时”和“已超时”显示
- 详情页 SLA 卡片显示：
  - 目标时长
  - 已耗时
  - 是否暂停
  - 是否超时
- 新增风险工单视图
- 新增主管基础风险面板

## 7.5 风险点

- SLA 扫描任务需兼容 SQLite 与 MySQL
- 风险状态如果仅靠前端临时计算，容易和后端不一致
- 如果提醒机制未规划清楚，用户可能知道“超时了”但不知道怎么处理

## 7.6 验收标准

1. 工单超时后能自动进入风险状态
2. 主管可快速识别超时队列
3. SLA 风险可以在列表和详情页明确展示
4. 超时事件有完整留痕

## 8. Iteration 5：自动化规则与快捷动作

## 8.1 目标

让系统开始减少重复人工操作。

## 8.2 范围

本期包含：

- 自动化规则第一版
- 常用快捷动作 / 宏第一版
- 高优先级自动关注
- 按来源 / 分类自动分配团队

本期不包含：

- 通用低代码规则引擎
- 跨多个对象的复杂流程编排

## 8.3 后端任务

- 新增规则配置模型或固定规则配置能力
- 工单创建后执行自动分派
- 工单命中规则后写事件日志
- 新增快捷动作 / 宏配置能力

## 8.4 前端任务

- 新增规则配置页或规则配置区域
- 工单详情页增加“套用快捷动作”入口
- 列表页批量操作可调用快捷动作

## 8.5 风险点

- 规则能力若设计过于通用，会明显拖慢迭代
- 快捷动作如果没有操作留痕，会影响审计
- 自动分派和人工分派规则冲突时，需要明确优先级

## 8.6 验收标准

1. 新工单可按规则自动落到合适团队
2. 高频标准处理可以通过快捷动作完成
3. 规则命中过程可追踪
4. 自动化不会绕过权限和事件留痕

## 9. Iteration 6：协作关系与 AI 辅助

## 9.1 目标

让工单系统从“内部工作台”进一步升级为“复杂问题处理平台”。

## 9.2 范围

本期包含：

- 工单关系
  - 重复工单
  - 父子工单
  - 相关工单
- 协作能力第一版
- AI 转单摘要
- AI 分类建议
- AI 优先级建议
- AI 回复草稿

## 9.3 后端任务

- 扩展 `TicketRelation` 的实际产品能力
- 工单详情支持查询和维护关联关系
- 接入 AI 服务为转工单和回复场景提供建议结果

## 9.4 前端任务

- 工单详情侧栏增加关联工单编辑能力
- 会话转工单弹窗增加 AI 摘要建议
- 回复区增加 AI 草稿建议入口

## 9.5 风险点

- AI 建议如果质量不稳定，容易影响客服信任
- 协作能力设计过重会侵入工作台主流程
- 重复工单识别要控制误判风险

## 9.6 验收标准

1. 可建立和查看工单关系
2. 会话转工单支持自动摘要建议
3. 客服可一键采纳或编辑 AI 草稿
4. 协作与 AI 能力不会破坏现有主流程

## 10. 推荐推进顺序

推荐按以下顺序推进：

1. 配置中心与结构化字段
2. 列表页队列化
3. 详情页重构
4. 数据权限
5. SLA 风险运营
6. 自动化规则
7. 快捷动作
8. 协作关系
9. AI 辅助

原因：

- 配置中心决定字段基础
- 列表和详情决定日常体验
- 数据权限决定真实可上线
- SLA 和规则决定运营效率
- 协作和 AI 是增益项，不适合抢跑在底座前面

## 11. 人员分工建议

建议最少按以下角色组织：

- 产品负责人
  - 负责需求边界、优先级、验收
- 后端负责人
  - 负责模型、接口、权限、SLA 任务
- 前端负责人
  - 负责工作台页面和交互拆分
- 联调负责人
  - 负责接口契约、字段一致性和回归

如果人力更少，也应至少明确：

- 谁负责底层模型和接口
- 谁负责工作台页面
- 谁负责最终验收

## 12. 发布策略

建议采用“每个迭代独立上线”的方式：

- Iteration 1 可以先上线配置中心与表单升级
- Iteration 2 可以先上线列表改版
- Iteration 3 可以再上线详情页改版和数据权限
- Iteration 4 再上线 SLA 风险能力

不建议等待所有阶段都完成后一次性发大版本，否则周期太长，反馈太慢。

## 13. 关键风险清单

当前最需要关注的风险包括：

- 过早做通用规则引擎，导致整体节奏失控
- 数据权限方案不清晰，导致后期返工
- 工作台页面堆太多信息，造成体验恶化
- SLA 风险逻辑在前后端出现不一致
- 历史工单对新字段和新配置兼容不足
- AI 建议介入过早，影响主流程稳定性

## 14. 当前建议结论

从当前阶段看，最值得优先推进的是：

- Iteration 1
- Iteration 2
- Iteration 3

也就是：

- 先把配置中心补齐
- 再把列表页改造成真正的队列页
- 再把详情页改造成真正的处理页

做到这一步后，工单系统就会从“有功能”进入“能真正承载内部客服异步处理”。

## 15. 总结

工单系统的建设不应以“补多少功能点”为目标，而应以“是否形成稳定处理闭环和管理闭环”为目标。

因此整个迭代计划要坚持一条主线：

- 先让数据结构标准化
- 再让页面承载工作流
- 再让系统具备运营能力
- 最后再叠加协作与 AI

按这个顺序推进，整体返工最少，价值释放也最稳定。
