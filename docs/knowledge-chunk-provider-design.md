# 知识库 Chunk Provider 设计方案

本文档用于整理知识库 RAG 分块策略的 provider 化设计方案，目标是将分块能力抽象为可插拔实现，使系统可以根据知识类型、文档结构和精度要求选择不同的 chunk provider。

## 1. 设计目标

当前系统使用固定字符长度分块，主要问题是：

- 不理解文档结构
- 容易破坏语义边界
- 对 FAQ、步骤说明、表格型知识效果较差
- 不利于后续持续优化和做 A/B 对比

因此建议将分块能力设计为 provider 模式，支持多套实现并允许按配置切换。

目标包括：

1. 支持多种 chunk 策略并存
2. 支持不同知识库选择不同 provider
3. 支持后续逐步引入更高级的分块算法
4. 支持对不同 provider 的效果进行对比评估

## 2. 推荐提供的 Provider 列表

建议第一阶段正式提供以下 4 个 provider：

1. `fixed`
2. `structured`
3. `faq`
4. `semantic`

同时建议预留一个第二阶段可扩展 provider：

5. `hierarchical`

其中：

- `fixed` 用于兜底和基线对比
- `structured` 建议作为默认 provider
- `faq` 用于客服问答类文档
- `semantic` 用于高精度场景
- `hierarchical` 作为后续高级检索能力演进方向

## 3. Provider 总体设计原则

### 3.1 抽象原则

每个 provider 只负责一件事：

- 输入原始文档内容
- 结合文档元信息和配置
- 输出标准化 chunk 列表

provider 不负责：

- embedding 生成
- 向量入库
- 检索排序
- 回答生成

### 3.2 标准输出

无论哪种 provider，最终都建议输出统一 chunk 结构，至少包括：

- `title`
- `content`
- `chunkType`
- `sectionPath`
- `charCount`
- `tokenCount`
- `metadata`

这样后续的 embedding、检索、重排、上下文构建都可以复用。

## 4. Provider 方案详解

---

## 4.1 `fixed` Provider

### 4.1.1 定位

`fixed` 是最简单、最稳定、最容易实现的一种分块方式，按固定长度将文本切成多块，并支持一定 overlap。

它的主要价值不是“效果最好”，而是：

- 作为兜底 provider
- 作为对照基线
- 作为低复杂度场景的默认保底实现

### 4.1.2 原理

基本流程：

1. 将文档转成纯文本
2. 按固定字符数或 token 数切分
3. 相邻 chunk 之间保留 overlap

示例：

```text
chunk1: 1-300 token
chunk2: 260-560 token
chunk3: 520-820 token
```

### 4.1.3 优点

- 实现最简单
- 行为稳定、可预测
- 适合做基线对比
- 不依赖复杂文档解析

### 4.1.4 缺点

- 不理解标题、段落、列表、表格结构
- 容易把多个主题切进同一块
- 容易把一个完整语义单元切断
- RAG 精度通常最弱

### 4.1.5 适用场景

- 纯文本
- 文档结构非常弱
- 开发初期兜底
- 回归测试和 A/B 对比

### 4.1.6 是否建议默认使用

不建议作为业务默认 provider，但必须保留。

---

## 4.2 `structured` Provider

### 4.2.1 定位

`structured` 是最适合当前系统的默认 provider。

它的目标是先识别文档结构，再基于结构切块，保证 chunk 尽量不破坏标题和段落边界。

这是当前企业知识库、帮助中心、产品文档中最常见也最成熟的一类方案。

### 4.2.2 原理

核心思想是“结构优先”：

1. 先解析文档结构
2. 识别标题、段落、列表、表格、代码块等元素
3. 按 section 组织内容
4. section 过长时再按段落或句子二次切分

Markdown 中重点识别：

- `#` `##` `###`
- 段落
- 列表
- 代码块
- 表格

HTML 中重点识别：

- `h1-h6`
- `p`
- `ul/ol/li`
- `table/tr/td/th`
- `pre/code`

### 4.2.3 优点

- 最大程度保留文档结构
- 比固定切分更符合人工阅读语义
- 对操作说明、产品文档、帮助中心效果明显更好
- 工程复杂度适中，收益非常高

### 4.2.4 缺点

- 需要做 HTML/Markdown 结构解析
- 对非常随意的非结构化文本收益有限
- 对语义突变检测不如 semantic provider 精细

### 4.2.5 适用场景

- 帮助中心文章
- 产品说明文档
- 操作教程
- 知识库正文
- 含标题层级的 Markdown / HTML 文档

### 4.2.6 是否建议默认使用

建议作为系统默认 provider。

### 4.2.7 推荐切分原则

建议遵循以下规则：

1. 标题作为 section 边界
2. 不跨 section 合并内容
3. 列表优先整体保留
4. 表格优先整体保留
5. 代码块单独成块
6. 超长 section 再按段落或句子切分

---

## 4.3 `faq` Provider

### 4.3.1 定位

`faq` 是客服系统非常值得单独支持的 provider。

因为 FAQ 类知识和普通长文档的结构差异很大。如果仍然使用普通段落分块，问答对关系很容易被破坏，影响召回质量。

### 4.3.2 原理

核心思想是“问答对优先”：

1. 识别 FAQ 问题与答案边界
2. 一个问答对尽量形成一个 chunk
3. 如果答案太长，再在答案内部继续细分
4. 保留问题文本作为 chunk 的重要语义头部

常见识别形式包括：

- `Q:` / `A:`
- `问：` / `答：`
- 标题 + 简短解释
- 有明显问题句式的短标题段

建议 embedding 文本组织形式：

```text
问题：如何申请退款？
回答：进入订单页后点击申请退款，等待审核...
```

### 4.3.3 优点

- 对客服问答类知识非常友好
- 用户问题与 chunk 语义匹配度高
- 常见短问短答场景下召回效果通常明显优于普通分块

### 4.3.4 缺点

- 只适合 FAQ 型知识
- 如果文档不规范，FAQ 识别规则需要不断补强
- 不适合大段连续说明文

### 4.3.5 适用场景

- 常见问题库
- 工单标准回复
- 知识问答条目
- 客服短问题知识沉淀

### 4.3.6 是否建议默认使用

不建议全局默认，但建议作为重要专用 provider 提供。

### 4.3.7 推荐切分原则

1. 尽量保持一问一答为同一 chunk
2. 问题内容必须进入 embedding 文本
3. 答案过长时再按段落切分
4. FAQ chunk 类型明确标注为 `faq`

---

## 4.4 `semantic` Provider

### 4.4.1 定位

`semantic` 是高精度型 provider，目标是在文档内部找到更自然的语义边界，而不是只依赖标题或固定长度。

它适合后续精度优化阶段引入。

### 4.4.2 原理

核心思想是“语义断点切分”：

1. 先按句子切分
2. 计算相邻句子或句子窗口的 embedding
3. 判断语义相似度变化
4. 在语义变化明显的位置切块

例如：

- 前几句在讲“退款流程”
- 后几句突然转向“退款失败原因”
- 在语义明显跳变处切出新 chunk

### 4.4.3 优点

- 更容易保留完整语义单元
- 对长段落、主题跳转频繁的文档更友好
- 精度潜力高

### 4.4.4 缺点

- 实现复杂度高
- 索引构建成本高，需要额外 embedding 或语义比较
- 调参更复杂
- 对文档质量、句子切分质量更敏感

### 4.4.5 适用场景

- 长文档
- 多主题说明文
- 结构不稳定但语义较连贯的内容
- 后续高精度知识库

### 4.4.6 是否建议默认使用

不建议第一阶段作为默认 provider，但建议纳入正式方案中。

### 4.4.7 推荐使用方式

建议：

- 第一阶段保留接口和设计
- 第二阶段在少量知识库试点
- 先做离线评估，再扩大使用范围

---

## 4.5 `hierarchical` Provider

### 4.5.1 定位

`hierarchical` 更准确地说不是单纯“分块方法”，而是“分块与检索联动策略”。

它适合第二阶段升级，不建议第一阶段就直接落地。

### 4.5.2 原理

核心思想是“父子块”：

1. 先按 section 切出父块
2. 再在父块内切出子块
3. 向量检索时召回子块
4. 给 LLM 上下文时回溯父块或相邻块

例如：

- 子块负责精准召回
- 父块负责补足上下文

### 4.5.3 优点

- 兼顾召回精度与上下文完整性
- 非常适合长文档
- 是很多成熟 RAG 系统的常见升级方向

### 4.5.4 缺点

- 实现复杂度高
- 数据结构和检索逻辑都要改
- 评估和调参成本较高

### 4.5.5 适用场景

- 长篇产品文档
- 手册类知识库
- 多章节知识文档
- 高质量客服辅助场景

### 4.5.6 是否建议第一阶段实现

不建议第一阶段强上，但建议预留演进方向。

---

## 5. 建议的默认落地顺序

从投入产出比看，建议按下面顺序落地：

### 第一阶段

1. `fixed`
2. `structured`
3. `faq`

其中：

- `structured` 作为默认 provider
- `faq` 作为客服问答类知识专用 provider
- `fixed` 作为兜底与回归基线

### 第二阶段

4. `semantic`
5. `hierarchical`

## 6. 推荐的默认策略

建议系统中的默认 provider 策略如下：

### 6.1 系统默认

默认 provider 使用：

- `structured`

原因：

- 通用性强
- 对现有 HTML / Markdown 知识库最友好
- 改造成本低于 semantic
- 精度提升明显

### 6.2 特殊内容按类型切换

建议支持文档或知识库级别切换 provider：

- FAQ 类知识使用 `faq`
- 普通帮助文档使用 `structured`
- 对照实验或兜底使用 `fixed`
- 高精度试验使用 `semantic`

## 7. Provider 设计建议

建议将 provider 设计为统一接口，例如：

```go
type ChunkProvider interface {
    Name() string
    Supports(contentType string) bool
    Chunk(ctx context.Context, req ChunkRequest) ([]ChunkResult, error)
}
```

### 7.1 ChunkRequest 建议字段

- `knowledgeBaseID`
- `documentID`
- `documentTitle`
- `contentType`
- `content`
- `options`

### 7.2 ChunkResult 建议字段

- `title`
- `content`
- `chunkType`
- `sectionPath`
- `charCount`
- `tokenCount`
- `metadata`

## 8. 与当前系统的适配建议

结合当前系统实现，建议第一阶段做如下适配：

1. 将现有 `chunkText` 改造成 provider 调度入口
2. 默认接入 `structured` provider
3. 保留 `fixed` 作为现有逻辑兼容方案
4. 给 `KnowledgeChunk` 增加必要元数据字段
5. 在向量 payload 中同步写入 chunk 元数据

## 9. 最终建议

如果只考虑当前阶段最务实的方案，建议正式支持以下 4 个 provider：

1. `fixed`
2. `structured`
3. `faq`
4. `semantic`

并将：

- `structured` 作为默认 provider
- `faq` 作为客服问答专用 provider
- `fixed` 作为兜底和基线
- `semantic` 作为后续高精度增强 provider

`hierarchical` 建议作为第二阶段演进方向预留，不必第一阶段就强行落地。
