# 工单工作台改版设计

本文档用于定义工单工作台的改版方案，目标是在现有工单模块已经具备基础创建、指派、状态流转、回复与备注能力的前提下，将当前“可用但偏 CRUD”的工单页面升级为真正适合客服和主管高频使用的工作台。

本文档重点覆盖：

- 工单列表页改版
- 工单详情页改版
- 会话与工单联动优化
- 列表与详情的接口诉求
- 前端组件拆分建议
- 第一阶段验收标准

## 1. 背景

### 1.1 当前现状

当前系统已经具备以下工单能力：

- 工单列表
- 工单详情
- 新建工单
- 会话转工单
- 指派 / 转派
- 状态流转
- 公开回复 / 内部备注
- 关闭 / 重开
- 基础 SLA 展示
- 关注人 / 事件日志

但从实际产品体验看，当前工单模块仍更接近“管理后台中的一个业务页面”，而不是“客服工作台中的处理主战场”。

当前主要问题包括：

- 列表页偏普通表格页，缺少明显的队列感
- 详情页信息堆叠，主次不够清晰
- 主管视角与客服视角没有很好区分
- 列表和详情之间缺少顺滑的高频处理路径
- 会话和工单已经建立关联，但上下文连续性仍不够强
- SLA 虽有字段和记录，但列表页和详情页尚未把风险感知做出来

### 1.2 为什么要先做工作台改版

在当前阶段，不考虑客户端闭环的前提下，工单系统能否真正用起来，主要取决于两个页面：

- 工单列表页
- 工单详情页

如果这两个页面仍然是典型 CRUD 形态，那么：

- 客服很难快速处理大量异步问题
- 主管很难识别队列风险
- 后续即使补了分类、SLA、自动化和 AI，价值也很难被放大

因此，工单工作台改版是工单系统从“有模块”走向“能运营”的核心步骤。

## 2. 设计目标

本次改版的目标如下：

- 让客服能快速找到自己要处理的工单
- 让主管能快速识别待分配、待处理和已超时工单
- 让工单详情页成为完整的处理主战场
- 让会话转工单和工单回看会话更自然
- 让后续分类、SLA、批量操作、自动化、AI 能力有合适的页面承载

## 3. 用户角色与使用目标

## 3.1 一线客服

关注点：

- 我的工单
- 哪些工单优先处理
- 如何快速回复、备注、转派、关闭
- 如何快速从会话转工单
- 如何减少重复操作

## 3.2 组长 / 主管

关注点：

- 哪些工单无人负责
- 哪些工单已超时或即将超时
- 团队当前负载如何
- 工单是否卡在待客户反馈或待内部处理中
- 是否需要干预分配和升级

## 4. 改版范围

### 4.1 本期包含

- 工单列表页改版
- 工单详情页改版
- 会话页的工单联动增强
- 列表页批量操作第一版
- 详情页 SLA / 来源会话 / 关联工单区域优化

### 4.2 本期不包含

- 客户端工单中心
- 客户可见回复链路
- 通用规则引擎
- 工单宏完整体系
- AI 自动处理闭环
- 多租户 / 多品牌工作台切换

## 5. 工单列表页设计

## 5.1 页面定位

工单列表页不是单纯的“查数据页”，而是“队列页”。

客服进入列表页的目标不是浏览数据，而是：

- 快速看到当前要处理的工单
- 用最少操作进入处理
- 必要时做批量处理

主管进入列表页的目标是：

- 快速识别异常队列
- 发现超时和待分配问题
- 对队列进行分配和干预

## 5.2 页面结构

建议结构如下：

```text
顶部：页面标题 + 快捷视图卡片 + 新建工单按钮
筛选区：关键词 / 状态 / 分类 / 优先级 / 严重度 / 团队 / 处理人 / 来源
主体区：工单表格
表格上方：批量操作条（有选中项时出现）
底部：分页
```

## 5.3 快捷视图

第一阶段建议提供以下固定视图：

- 全部工单
- 我的工单
- 待分配
- 待客户反馈
- 待内部处理
- 已超时

说明：

- `全部工单` 用于全局浏览
- `我的工单` 用于客服日常入口
- `待分配` 用于主管队列分派
- `待客户反馈` 用于检查卡住的工单
- `待内部处理` 用于检查内部协作积压
- `已超时` 用于风险处置

后续可以再升级为“保存视图”。

## 5.4 筛选项

建议支持以下筛选：

- 关键词
- 状态
- 分类
- 优先级
- 严重度
- 当前处理团队
- 当前处理人
- 来源
- 是否已关闭
- 是否我关注

关键词建议匹配：

- 工单号
- 标题
- 描述摘要

## 5.5 排序策略

第一阶段建议支持：

- 默认按更新时间倒序
- 可按优先级排序
- 可按 SLA 风险排序
- 可按创建时间排序

其中“按 SLA 风险排序”对主管非常重要。

## 5.6 表格列设计

建议默认列如下：

- 工单
  - 工单号
  - 标题
- 客户
- 分类
- 优先级
- 状态
- 当前处理人
- 当前团队
- SLA 风险
- 更新时间
- 操作

说明：

- “工单”列可同时展示标题和工单号，减少横向空间占用
- `SLA 风险` 列建议明确显示：
  - 正常
  - 即将超时
  - 已超时

## 5.7 行级操作

建议第一阶段支持：

- 查看详情
- 指派
- 改状态
- 关闭
- 关注 / 取消关注

不建议把过多复杂操作堆在表格行上，否则可读性会迅速下降。

## 5.8 批量操作

第一阶段建议支持：

- 批量指派
- 批量改状态
- 批量关注

第二阶段再考虑：

- 批量应用宏
- 批量关闭
- 批量加标签

## 5.9 空状态

建议明确区分：

- 当前系统暂无工单
- 当前筛选条件下无结果

空状态需给出清晰行动入口：

- 清空筛选
- 新建工单

## 6. 工单详情页设计

## 6.1 页面定位

工单详情页是工单处理主战场。

客服在详情页应完成：

- 理解问题背景
- 查看上下文
- 回复客户
- 添加内部备注
- 修改状态
- 指派 / 转派
- 关闭 / 重开
- 回看事件过程

主管在详情页应完成：

- 判断工单卡点
- 看处理责任和流转痕迹
- 快速干预分派和状态

## 6.2 页面布局

建议采用双栏布局：

主区域：

- 头部摘要
- 回复 / 备注区
- 时间线区域

右侧侧栏：

- 基础信息
- 客户信息
- 指派信息
- SLA 信息
- 来源会话
- 关联工单

## 6.3 头部区域

建议展示：

- 工单号
- 标题
- 状态 badge
- 优先级 badge
- 严重度 badge
- 分类
- 当前处理人
- 当前团队
- SLA 风险标签

头部操作区建议放：

- 分配处理人
- 变更状态
- 关注 / 取消关注
- 刷新
- 关闭 / 重开

## 6.4 回复与备注区域

当前详情页已具备公开回复和内部备注能力，但建议进一步强化：

- 明确区分“回复客户”和“内部备注”
- 后续支持模板入口
- 保持编辑区始终显眼，避免深层交互
- 提交成功后自动刷新时间线

## 6.5 时间线区域

建议改为 tab 切换：

- 回复记录
- 事件记录

原因：

- 当前把 comments 和 events 纵向堆叠，页面较重
- 客服高频看的是“人和人的沟通记录”
- 主管排查问题时更常看“状态流转事件”

第一阶段建议：

- 默认显示“回复记录”
- `事件记录` 独立成 tab

## 6.6 侧栏区域

### 基础信息

展示：

- 工单号
- 状态
- 分类
- 优先级
- 严重度
- 创建时间
- 更新时间
- 截止时间
- 重开次数

### 客户信息

展示：

- 客户名称
- 手机号
- 邮箱

### 指派信息

展示：

- 当前处理人
- 当前团队
- 关注人

### SLA 信息

展示：

- 首响 SLA
- 解决 SLA
- 当前状态
- 已耗时
- 是否暂停
- 是否超时

### 来源会话

展示：

- 来源会话 ID
- 会话标题
- 最近消息摘要
- 会话当前状态
- 前往会话按钮

### 关联工单

展示：

- 相关工单列表
- 后续支持重复工单 / 父子工单关系

## 6.7 详情页内编辑

建议第一阶段支持“轻量编辑”：

- 分类
- 优先级
- 严重度
- 处理人
- 团队

不建议所有属性都通过“回列表页点编辑弹窗”完成。

## 7. 会话与工单联动设计

## 7.1 会话页转工单入口

当前会话页已具备“转工单”入口，这个方向应保留。

建议继续强化：

- 会话头部操作区保留“转工单”
- 信息面板保留“关联工单”区域
- 转工单成功后可直接跳详情或继续留在会话

## 7.2 会话侧关联工单展示

当前信息面板中已能展示关联工单列表，后续建议增强为：

- 显示关联工单数量
- 高亮最近一张工单的状态
- 支持“查看更多”
- 若存在已超时工单，可提示风险

## 7.3 工单侧来源会话展示

工单详情页不应只展示一个 `conversationId`，还应给处理人足够上下文：

- 会话主题
- 最近消息摘要
- 当前会话状态
- 前往会话工作台按钮

这样跨班次和跨角色交接时，上下文才完整。

## 8. 前端组件拆分建议

建议新增或重构以下组件：

- `ticket-filter-bar.tsx`
- `ticket-list-table.tsx`
- `ticket-batch-action-bar.tsx`
- `ticket-detail-header.tsx`
- `ticket-comment-timeline.tsx`
- `ticket-event-list.tsx`
- `ticket-side-panel.tsx`
- `ticket-sla-card.tsx`
- `ticket-related-conversation-card.tsx`
- `ticket-related-list.tsx`

拆分目标：

- 降低页面文件复杂度
- 明确职责边界
- 方便后续加入批量操作、模板、AI 辅助等能力

## 9. 后端接口诉求

为支撑新版工作台，后端需要满足以下诉求。

## 9.1 列表接口

列表接口需支持：

- 更多筛选条件
- 更多 summary 统计
- 返回分类名称
- 返回 SLA 风险信息
- 支持批量操作接口

## 9.2 详情接口

详情接口需聚合：

- 工单基础信息
- 客户信息
- SLA 记录
- 关注人
- 评论记录
- 事件记录
- 分类名称
- 来源会话摘要
- 关联工单

## 9.3 批量接口

第一阶段建议补充：

- 批量指派
- 批量改状态
- 批量关注

## 10. 数据与交互原则

改版过程中应遵循以下原则：

- 列表页优先解决“快速找到和快速处理”
- 详情页优先解决“完整理解和完整处理”
- 重要状态必须可一眼识别
- SLA 风险信息必须具备明显视觉提示
- 列表与详情要形成清晰分工，避免职责混乱
- 不把所有字段一股脑塞进详情页，保持主次清晰

## 11. 第一阶段验收标准

以下条件满足后，可视为工作台改版第一阶段完成：

1. 列表页具备清晰的队列视图
2. 列表页支持分类、状态、优先级、团队、处理人等筛选
3. 列表页支持批量操作第一版
4. 详情页头部和侧栏信息层次清晰
5. 详情页支持回复、备注、指派、改状态、关闭、重开
6. 详情页可清楚查看 SLA 信息
7. 详情页可清楚查看来源会话信息
8. 会话页和工单页之间跳转顺畅
9. 客服能把工单详情页作为主要处理场所
10. 主管能通过列表页快速识别风险工单

## 12. 后续演进方向

工作台第一阶段改版完成后，可继续演进：

- 保存视图
- 宏 / 快捷动作
- SLA 风险提醒
- 批量关闭
- 工单关联关系
- AI 摘要
- AI 回复建议
- AI 分类建议
- 自动分派结果提示

## 13. 总结

工单工作台改版的重点不是“把页面做得更复杂”，而是让页面真正服务于客服处理流程和主管管理流程。

列表页要变成队列页，详情页要变成处理页，会话和工单要形成完整上下文链路。只有这样，后续分类配置、SLA 运营、自动化规则和 AI 辅助能力才能真正发挥价值。
